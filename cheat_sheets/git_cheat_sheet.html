Git Cheat Sheet

Introduction

This cheat sheet contains a list of all the git commands I think are useful to know. Here is a table of content, for easier navigation:
  • Basics
  • Git Branches
  • Git Diff
  • Stashing
  • Undoing Changes & Time Traveling

Basics

Initialise a repo

    git init
note: This command must be ran inside the directory we want to initialise a repo in.

Stage changes

  • git add <file_name>
  • git add *
    note: The second command allows us to add all modified, deleted and added files at once.

Commit

  • git commit
    note: After running this command, Git will open a window in our default editor to allow us to enter the commit message. 
  • git commit -m "<message>"
    note: With this variation, we specify the commit message directly in the terminal.

Stage all changes and commit (at once)

    git commit -am "<message>"
note: Here, all changes will be staged and commited at once! This is equivalent to commiting after running git add *, just faster to type.

See repostory status

   git status

See commits history

  • git log
  • git log --oneline
    note: The second command prints a shortened version of the commits hash and messages with one commit per line. It's useful to see only the essential, when commits messages are really long.

Modify the last commit

    git commit --amend
note: This command only allows us to modify the last commit. See further down for commands that allow to modify an arbitrary commit.
note: If we want to include an additional file in the commit, we first stage the file and then run this command. If we just want to modify the commit message of the last commit, we simply run this command and git will ask us for a new commit message.

Git Branches

See what branch we are currently on

    git branch

Create a new branch (without switching to it)

    git branch <branch_name>
note: The location in the commits tree where the new branch is created depends on where the current HEAD is!

Switch to another branch

  •     git switch <branch_name>
  •     git checkout <branch_name>
note: git switch is a new command introduced to lessen the use of the git checkout command which does too many things.

Create a new branch and switch to it

    git switch -c <branch_name>

Delete a branch

  •     git branch -d <branch_name>
  •     git branch -D <branch_name>
note: We must be on another branch than the one we are trying to delete when running this command!
note: The second option, using -D, forces the delete even if the branch is not merged.

Rename a branch

    git branch -m <new_branch_name>
note: We must be on the branch we want to rename when running this command.

Merge <branch_2> into <branch_1>

    git merge <branch_2>
note: We must be on <branch_1> when running this command.
note: After running this command, a new commit with the changes of <branch_2> will be created on <branch_1> but <branch_2> will stay separated and will not be modified at all.
note: If there are merge conflicts, the files with the conflicts will be modified by git to put into evidence the conflicts. A windows showing these conflicts in the default editor configured in git will then be opened to allow the user to choose which parts of the conflicting code to keep and which part to throw away. Once this is done, the user can add and commit the results which will create the merge commit.

Git Diff

General note: all the diff commands can be applied (restricted) to a specific file by passing the file name after the command (e.g. git diff <my_file>).

See the difference between the unstaged changes in the working directory and the staged changes

    git diff
note: This is a comparison between the working directory and the staging area.

See the difference between the (staged and unstaged) changes in the working directory and the last commit

    git diff HEAD
note: This is a comparison between the [unstaged working directory + staging area] and the last commit.

See the difference between the staged changes in the working directory and the last commit

  •     git diff --staged
  •     git diff --cached
note: This allows us to see all the changes staged since the last commit.
note: This is a comparison between the staging area and the last commit.
note: The two commands are equivalent.

See the difference between the last commits of two branches

  •     git diff <branch_1> <branch_2>
  •     git diff <branch_1>..<branch_2>
note: This is a comparison between the last commits of two branches.
note: The two commands are equivalent.

See the difference between two commits on the same branch

  •     git diff <commit_1_hash> <commit_2_hash>
  •     git diff <commit_1_hash>..<commit_2_hash>
note: This is a comparison between two commits on the same branch.
note: The two commands are equivalent.

Stashing

Stash all uncommitted changes

    git stash
note: This will stash both staged and unstaged changes.
note: This will revert the working directory to the state of the last commit.

Remove the most recently stashed changes from the stash & apply them to the working directory

    git stash pop
note: we can pop the changes on the stash on ANY branch, even a different branch that the one the changes were originally stashed from! For example, I can stash the changes on <branch_1> and then pop them on <branch_2>.

Apply the most recently stashed changes to the working directory without removing them from the stash

    git stash apply
note: This does exactly the same thing than git stash pop but without removing the changes from the stash.
note: This command can be used in the same way as git stash pop - see the notes for this command.

List the changes stored in the stash

    git stash list
note: The git stash is a stack (L.I.F.O.).
note: This shows the elements along with their indices, e.g. stash@{0}, stash@{1}, stash@{2}, ..., etc. These indices can be used to target specific elements of the stash (i.e. specific stored changes). For example, we can run the command git stash apply stash@{2} to apply the changes corresponding to the 3rd element of the stash to our working directory while keeping them in the stash (because we are using apply and not pop).

Deleting a specific change from the stash

    git stash drop stash@{<index>}
note: Here <index> is the index in the stash of the change we want to drop. For example, git stash drop stash@{2} to drop the 3rd element of the stash.

Clearing (deleting/emptying) the entire stash

    git stash clear
